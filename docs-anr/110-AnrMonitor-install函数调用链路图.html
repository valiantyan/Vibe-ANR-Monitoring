AnrMonitor.install() 完整函数阅读路线图

这份 HTML 面向第一次阅读 SDK 源码的新人：不是展示散点节点图，而是从 AnrMonitor.install() 开始，沿着真实代码执行路径，把安装、消息采集、Watchdog、ANR 现场采集、归因、JSON 写入、上传重试和卸载链路串成可阅读的函数路线。

先记住这 4 件事

AnrMonitor.install() 不是 SDK 的全部工作，只是把监控能力安装到进程里。真正生成 ANR JSON 的链路，会在 Watchdog 线程发现主线程心跳超时后异步触发。

同步调用 构造对象 开启异步线程/回调 报告生成链路 停止卸载链路
入口在哪里 api/AnrMonitor.kt 的 install() 是公开入口。
启动了什么 安装 Looper Printer、启动 Watchdog、可选启动上传重试线程。
JSON 何时生成 Watchdog 判断主线程心跳超时后，调用 captureSuspectAnr()。
怎么继续阅读 按本页 1 到 8 段顺序读，先主干，再看每条分支。

1. 公开安装入口：先弄懂 SDK 怎么被装进去

同步调用 
AnrMonitor.install(...)
  ├─ activeSession 非空：return existingSession
  ├─ AnrMonitorRuntime(...)
  └─ installRuntimeLocked(config, runtimeHandle)
函数 所在文件 职责 下一步读哪里
AnrMonitor.install() api/AnrMonitor.kt 公开安装入口；用 @Synchronized 保护单例安装，避免重复启动多个 Watchdog 或 Looper Printer。 看到没有旧 session 后，继续读 AnrMonitorRuntime(...) 构造。
activeSession api/AnrMonitor.kt 进程内唯一活跃会话；重复 install 直接返回旧 session。 理解为什么重复调用不会生成多套采集器。
  • 读完这一段，要知道 SDK 的公开入口只有一个，重复安装会被短路。
  • 这里还没有生成 JSON，也没有立刻采集 ANR。

2. Runtime 构造依赖：看 SDK 准备了哪些能力

构造对象 
AnrMonitorRuntime(...)
  ├─ clock / sdkSelfMonitor / sanitizer
  ├─ historyBuffer / stackCollector / slowMessageStackSampler
  ├─ timelineCollector
  ├─ pendingSnapshotter / barrierEvidenceCollector / binderBlockClassifier
  ├─ threadCpuSnapshotter / checktimeMonitor / environmentSnapshotter
  ├─ anrInfoCollector
  ├─ reportAssembler / reportEncoder / localWriter
  ├─ reportRetryQueue / reportRetryDispatcher

对象 所在文件 职责 是否启动线程
MainLooperTimelineCollector collector/looper/MainLooperTimelineCollector.kt 维护当前消息、历史消息、慢消息栈采样，是 JSON 中主线程消息时间线的来源。 否，只是被构造。
PendingQueueSnapshotter collector/pending/PendingQueueSnapshotter.kt 疑似 ANR 时读取 MessageQueue pending 消息，用于识别消息堆积和 Sync Barrier。 否。
BarrierEvidenceCollector collector/barrier/BarrierEvidenceCollector.kt 聚合 Barrier token 和 nativePollOnce 证据，辅助判断 nativePollOnce 是否被 Sync Barrier 挡住。 否。
AnrWatchdog collector/watchdog/AnrWatchdog.kt 后台心跳检测器；构造时把 onSuspectAnr 绑定到 captureSuspectAnr()。 构造时不启动，start() 才启动。
LocalAnrReportWriter reporter/local/LocalAnrReportWriter.kt 负责把完整报告写入 app 私有目录下的 anr-monitor-reports。 否。
  • 读完这一段，要知道 Runtime 是“编排器”，不是单一 collector。
  • 构造阶段只准备对象，真正接线在 AnrMonitorRuntime.start()。

3. 会话绑定与启动：install 同步阶段的核心

同步调用 
installRuntimeLocked(config, runtimeHandle)
  ├─ AnrMonitorSession(config, stopAction)
  ├─ runtime = runtimeHandle
  ├─ activeSession = session
  ├─ runtimeHandle.start()
  │   └─ AnrMonitorRuntime.start()
  │       ├─ MainLooperPrinterInstaller().install(timelineCollector)
  │       ├─ watchdog.start()
  │       └─ startUploadRetryLoop()
  └─ return session
函数 所在文件 职责 下一步读哪里
installRuntimeLocked() api/AnrMonitor.kt 创建 session，保存 runtime/session，然后启动 runtime。 AnrMonitorSession.stop() 和 AnrMonitorRuntime.start()。
AnrMonitorSession(...) api/AnrMonitorSession.kt 宿主拿到的会话句柄；保存配置和停止动作。 读 stop() 了解卸载路径。
AnrMonitorRuntime.start() internal/AnrMonitorRuntime.kt 真正接入监控能力：Looper Printer、Watchdog、上传重试线程。 先读 Looper Printer，再读 Watchdog。
  • 读完这一段，要知道 install 返回前只完成“安装与启动”，不等待 ANR 发生。
  • start() 有 config.enabled 和 isRunning 防重保护。

4. Looper 消息采集：JSON 里的 currentMessage/historyMessages 从哪里来

主 Looper 回调 
MainLooperPrinterInstaller.install(timelineCollector)
  ├─ requireLooper()
  ├─ readCurrentPrinterFromLooper(looper)
  ├─ chainedPrinter.println(line)
  │   ├─ timelineCollector.println(line)
  │   └─ previousPrinter?.println(line)
  └─ looper.setMessageLogging(chainedPrinter)

MainLooperTimelineCollector.println(line)
  └─ onLooperLog(line)
      ├─ LooperMessageParser.parse(line)
      ├─ createCurrentRecordStart(event)
      │   └─ startSlowSampling(start)
      └─ finishCurrentRecord(event)
          ├─ finishSlowSampling(seq)
          └─ historyBuffer.add(record)
函数 所在文件 职责 读代码时看什么
MainLooperPrinterInstaller.install() collector/looper/MainLooperPrinterInstaller.kt 把 SDK 的 Printer 安装到主 Looper，同时链式调用已有 Printer，降低冲突。 看 previousPrinter、chainedPrinter、InstallHandle。
onLooperLog() collector/looper/MainLooperTimelineCollector.kt 解析 Looper 开始/结束日志，分别进入“当前消息开始”和“历史消息结束”。 看 start 分支和 finish 分支怎么改变 currentRecordStart。
currentMessage() collector/looper/MainLooperTimelineCollector.kt Watchdog 捕获现场时读取当前未完成消息，并可能触发慢消息栈采样。 看 wallMs、cpuMs、sampleStackIds 如何计算。
historyMessages() collector/looper/MainLooperTimelineCollector.kt 返回环形缓冲区里的已完成历史消息。 看它为什么能帮你找到“ANR 前一个慢消息”。
  • 读完这一段，要能解释 JSON 的 mainThread.currentMessage 和 mainThread.historyMessages。
  • 这里是主线程消息自然回调，不是 Watchdog 线程主动抓取所有消息。

5. Watchdog 心跳检测：SDK 怎么发现疑似 ANR

后台线程 
AnrWatchdog.start()
  ├─ isRunning.compareAndSet(false, true)
  └─ Thread(::loop, "vibe-anr-watchdog").start()

AnrWatchdog.loop()
  ├─ recordChecktimeInterval(now)
  ├─ heartbeatState.isTimedOut(now)
  │   ├─ true: onSuspectAnr()
  │   └─ recovered: onRecovered()
  ├─ postHeartbeat()
  │   ├─ heartbeatState.markPosted(seq, postedUptimeMs)
  │   └─ mainHandler.post { heartbeatState.markHandled(seq) }
  └─ sleepInterval()
函数 所在文件 职责 下一步读哪里
AnrWatchdog.start() collector/watchdog/AnrWatchdog.kt 启动后台检测线程，重复调用不会创建多条线程。 loop()。
loop() collector/watchdog/AnrWatchdog.kt 周期性检查心跳是否超时，超时则回调 onSuspectAnr()。 postHeartbeat() 和 HeartbeatState。
postHeartbeat() collector/watchdog/AnrWatchdog.kt 向主线程投递一个心跳 Runnable；主线程执行到它，说明消息循环还能推进。 看主线程卡住时为什么 markHandled() 不会执行。
onSuspectAnr() 构造时绑定到 AnrMonitorRuntime.kt 函数引用，实际调用 AnrMonitorRuntime.captureSuspectAnr()。 进入现场采集链路。
  • 读完这一段，要知道 SDK 不是依赖系统 ANR 弹窗才工作，而是自己用心跳提前捕获疑似 ANR。
  • 如果主线程卡在 nativePollOnce、Binder、IO、CPU 忙等，心跳都可能无法及时处理。

函数 所在文件 职责 对应 JSON 内容
captureSuspectAnr() internal/AnrMonitorRuntime.kt Watchdog 超时后的入口；做限频和异常兜底。 决定是否产生本次报告。
shouldCapture() internal/AnrMonitorRuntime.kt 运行态和时间窗口限频，避免同一阻塞窗口疯狂生成报告。 影响是否有 JSON。
buildSnapshot() internal/AnrMonitorRuntime.kt 把所有采集器结果合成 AnrSnapshot。 event、mainThread、pendingQueue、barrierEvidence、binderBlock 等。
capturePendingQueue() internal/AnrMonitorRuntime.kt 按配置读取 MessageQueue pending 消息；关闭时返回 unavailable。 pendingQueue。
captureBarrierEvidenceSnapshot() internal/AnrMonitorRuntime.kt 采集 Sync Barrier token 和 nativePollOnce 增强证据。 barrierEvidence。
captureBinderBlockSnapshot() internal/AnrMonitorRuntime.kt 采集主线程 Binder 阻塞疑似和 Binder 线程辅助栈。 binderBlock。
  • 读完这一段，要能把 JSON 每个大节点对应回一个采集函数。
  • 如果某个节点 unavailable，要回到对应 captureXxx() 看配置开关或失败原因。

8. 停止与卸载：看 SDK 怎么释放资源

停止链路 
AnrMonitor.uninstall()
  ├─ activeSession?.stop()
  │   └─ AnrMonitorSession.stop()
  │       └─ stopAction()
  │           ├─ runtimeHandle.stop()
  │           │   └─ AnrMonitorRuntime.stop()
  │           │       ├─ watchdog.stop()
  │           │       ├─ looperPrinterConflictReporter.record(handle)
  │           │       ├─ looperPrinterHandle?.uninstall()
  │           │       ├─ markAnrRecovered()
  │           │       └─ stopUploadRetryLoop()
  │           └─ clearSessionIfCurrent(session)
  ├─ activeSession = null
  └─ runtime = null
函数 所在文件 职责 注意点
AnrMonitorSession.stop() api/AnrMonitorSession.kt 宿主主动停止入口；调用 runtime stop，并标记 session 不再运行。 重复 stop 会提前返回。
AnrMonitorRuntime.stop() internal/AnrMonitorRuntime.kt 停止 Watchdog，恢复 Looper Printer，清理去重状态，停止上传线程。 先停 Watchdog，再恢复 Printer，避免停止中继续采集。
InstallHandle.uninstall() collector/looper/MainLooperPrinterInstaller.kt 只有当前槽位仍是 SDK 安装的 Printer 时，才恢复 previousPrinter。 避免误覆盖 SDK 之后安装的第三方 Printer。
stopUploadRetryLoop() internal/AnrMonitorRuntime.kt 中断上传重试线程并清空引用。 上传关闭时这个线程本来就可能不存在。
  • 读完这一段，要知道 SDK 停止会释放后台线程和 Looper Printer。
  • 如果排查重复安装、重复报告、Printer 冲突，停止链路必须一起看。

按问题类型继续读哪个函数

想理解 nativePollOnce / Sync Barrier

先读 buildSnapshot() → capturePendingQueue() → captureBarrierEvidenceSnapshot()
再读 AttributionAnalyzer.analyzeBarrier() 和 hasStrongBarrierEvidence()
最后对照 JSON 的 pendingQueue、barrierEvidence、mainThread.stack。

想理解当前慢消息 / CPU 忙等

先读 MainLooperTimelineCollector.currentMessage()
再读 collectSlowStackSampleIfNeeded() 和 currentCpuMs()
最后对照 JSON 的 currentMessage.wallMs、cpuMs、stackSamples。

想理解消息风暴

先读 capturePendingQueue() 和 PendingQueueAnalyzer.analyze()
再读 AttributionAnalyzer.analyzeMessageStorm()
最后对照 JSON 的 pendingQueue.messages 同类 target 数量。

想理解 Binder / 跨进程阻塞

先读 captureBinderBlockSnapshot()
再读 BinderThreadStackCollector.capture() 和 BinderBlockClassifier.classify()
最后对照 JSON 的 binderBlock 和主线程栈中的 Binder frame。

这份图的边界

不是所有工具函数都展开。 例如 JSON 字符串转义、File 元信息转换、每个 data class 构造器没有逐一展开，因为它们不会帮助新人理解 SDK 主流程。

架构说明

架构层 边界职责 依赖方向
宿主 Demo App 负责安装 SDK、提供配置、触发可验证 ANR 场景。 只依赖 SDK Public API，不直接依赖 SDK 内部实现。
Public API 层 暴露安装、卸载、配置、监听器、上传器。 向内创建 Runtime，不暴露 collector 和 reporter 细节。
Runtime 编排层 统一启动采集器，接收 Watchdog 事件，组装快照，触发归因和写入。 向下依赖 collector、domain、reporter，是 SDK 内部主控层。
Collector 层 持续采集主 Looper 时间线；疑似 ANR 时采集 Pending、栈、CPU、环境、Barrier、Binder 证据。 输出领域模型，不直接决定根因。
Domain / Analyzer 层 定义稳定报告模型，按证据优先级输出主因、置信度和修复建议。 读取 AnrSnapshot，输出 AttributionResult。
Reporter 层 把报告编码成 JSON，落盘到私有目录，并支持保留策略和上传重试。 只消费 AnrReport，不反向影响采集和归因。

架构评审关注点

  • 依赖方向是单向的：App 到 Public API，Public API 到 Runtime，Runtime 到 Collector / Domain / Reporter。
  • 采集器只负责生产证据，归因逻辑集中在 AttributionAnalyzer，避免根因规则散落在采集代码里。
  • nativePollOnce 类问题不会只依赖主线程栈，而是通过 Pending 队列、Barrier token 和 poll 记录共同确认。
  • 报告出口层不参与根因判断，保证写入失败、上传失败不会改变 ANR 现场事实。

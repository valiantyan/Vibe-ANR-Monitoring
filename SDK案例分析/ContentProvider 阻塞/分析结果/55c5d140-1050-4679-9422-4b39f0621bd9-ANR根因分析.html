ANR 根因分析报告：ContentProvider 阻塞

分析对象：55c5d140-1050-4679-9422-4b39f0621bd9.json。这份报告的核心结论是：点击事件触发了主线程上的 ContentResolver.query()，Provider 的 query() 内部执行了阻塞等待，导致当前主线程消息长时间无法结束。

最终结论

根因主线程在处理按钮点击消息时，同步调用 BlockingContentProvider.query()，Provider 内部执行 Thread.sleep()，导致点击消息耗时达到 3488ms。

性质这是一次 ContentProvider 阻塞引发的当前消息慢处理，属于主线程同步等待类 ANR 风险。

最关键证据

mainThread.current.callbackClass 是 android.view.View$PerformClick，说明正在处理点击事件。

mainThread.stackFrames 出现 ContentResolver.query()、BlockingContentProvider.query() 和 Thread.sleep()，业务链路闭合。

排除项

binderBlock.suspected=false，没有足够证据说明是 Binder 互等。

barrierEvidence.alignedWithPendingBarrier=false 且 stuckTokens=[]，不是 Sync Barrier 泄漏主因。

6. 排除其他常见误判

1

不是 Sync Barrier 泄漏主因

barrierEvidence.alignedWithPendingBarrier=false，stuckTokens=[]，nativePollOnceRecords=[]。虽然 Pending 队头有疑似 Barrier，但没有形成“Barrier 与 nativePollOnce 对齐、token 长时间不释放”的证据链。

2

不是 Binder 互等主因

binderBlock.suspected=false，mainThreadInBinder=false，binderThreadWaitsMain=false。报告没有显示主线程卡在 Binder 调用，也没有显示 Binder 线程等待主线程。

3

不是 CPU 忙等主因

mainThread.current.cpuMs=0，栈顶是 Thread.sleep，不是循环计算或高 CPU 消耗。threadCpu.topThreads 中 CPU 较高的是 RenderThread，不是这次根因。

确认 barrierEvidence 和 binderBlock 都没有形成主因证据链，最终根因收敛到 Provider 查询阻塞。

8. 修复方案

短期修复

  • 不要在主线程同步调用可能阻塞的 ContentResolver.query()。
  • 如果必须查询 Provider，把查询移到后台线程，完成后再把轻量结果切回主线程刷新 UI。
  • 给 Provider 查询增加超时、取消和失败兜底，避免调用方无限等待。
  • 点击按钮后先禁用重复点击或展示加载状态，避免主线程已经卡住时继续堆积输入事件。

长期治理

  • Provider 的 query()、insert()、update()、delete() 内不要做长时间 I/O、网络请求、锁等待或主动睡眠。
  • 如果 Provider 内部依赖远端服务或跨进程资源，要设计缓存、降级和异步预热，避免 UI 线程请求时被拖住。
  • 对所有主线程 Provider 调用加代码扫描或 lint 约束，把风险前移到代码评审阶段。
  • 在 SDK 上报平台中把 ContentProvider 阻塞栈归类为独立场景，聚合统计调用方、Provider 类名和阻塞耗时。

修复后验证

  • 再次点击 Demo 的 ContentProvider 阻塞按钮，不应继续生成同类 CURRENT_MESSAGE_SLOW 报告。
  • 如果仍生成报告，主线程栈不应再出现 BlockingContentProvider.query 和 Thread.sleep。
  • mainThread.current.wallMs 应显著下降，Pending 队列中的 UI 消息不应持续积压。

9. 可写进问题单的根因描述

本次 ANR 风险由主线程点击事件内同步查询 ContentProvider 引起。SDK 报告显示当前主线程消息为 View$PerformClick，已执行 3488ms 且 cpuMs=0，主线程栈停在 ContentResolver.query() 调用链内，最终进入 BlockingContentProvider.query() 并执行 Thread.sleep()。Pending 队列中 UI 和 Watchdog 消息已被延后 3 秒以上。Barrier、Binder 和 CPU 忙等证据不足，因此根因应判定为 ContentProvider 查询逻辑阻塞主线程。

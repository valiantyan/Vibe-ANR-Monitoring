ANR 根因分析报告：Binder 跨进程阻塞

分析对象：305f0ab4-a4ed-4b81-9b0b-f0bcf2034f94.json。本次最可信的根因是主线程在按钮点击回调中发起同步 Binder 调用，远端进程长时间不返回，导致主线程无法继续处理输入、绘制和后续消息。

结论

根因主线程同步等待跨进程 Binder 返回。

业务入口是 BinderCrossProcessBlockScenario.run()，调用链进入 IRemoteBlockingService$Stub$Proxy.blockFor() 后卡在 BinderProxy.transactNative。

SDK 归因

attribution.primary = BINDER_BLOCK_SUSPECTED

confidence = MEDIUM

这是“疑似 Binder 阻塞”，因为端侧能看到主线程在 Binder transact，但当前 JSON 没有远端进程的堆栈。

排除项

不是 Sync Barrier 泄漏：stuckTokens = []，nativePollOnceRecords = []。

不是资源压力主因：内存、存储可用；checktime.maxDelayMs = 2。

一步一步定位根因

1

先看 SDK 摘要，不急着翻完整 JSON

打开 attribution，看到：

{
  "primary": "BINDER_BLOCK_SUSPECTED",
  "confidence": "MEDIUM",
  "evidence": [
    "main thread blocked in Binder transact",
    "local binder thread wait evidence unavailable",
    "android.os.BinderProxy.transactNative(Native Method)"
  ]
}

这一步得到排查方向：主线程可能卡在 Binder 同步调用，不是先怀疑 CPU 忙等、消息风暴或 Barrier。

2

确认当前卡住的是哪一条主线程消息

看 mainThread.current：

{
  "messageType": "looper_dispatch",
  "targetClass": "android.view.ViewRootImpl$ViewRootHandler",
  "callbackClass": "android.view.View$PerformClick@a0add70",
  "wallMs": 3436,
  "endUptimeMs": null
}

View$PerformClick 表示当前主线程正在执行按钮点击回调；endUptimeMs = null 表示这条消息还没执行完；wallMs = 3436 表示它已经卡住 3.4 秒。

3

顺着主线程栈找到业务代码

看 mainThread.stackFrames 前几行：

android.os.BinderProxy.transactNative(Native Method)
android.os.BinderProxy.transact(BinderProxy.java:641)
com.valiantyan.vibeanrmonitoring.IRemoteBlockingService$Stub$Proxy.blockFor(IRemoteBlockingService.java:98)

从上往下读：主线程卡在系统 Binder transact；继续往下可以看到业务入口是 MainActivity 的点击回调，进入 BinderCrossProcessBlockScenario.run()，再通过 RemoteBlockingServiceClient.blockRemote() 调用远端服务。

4

用 binderBlock 节点复核是否真有 Binder 证据

本次 binderBlock 的关键值：

{
  "available": true,
  "suspected": true,
  "mainThreadInBinder": true,
  "binderThreadWaitsMain": false,
  "mainThreadEvidence": [
    "android.os.BinderProxy.transactNative(Native Method)",
    "android.os.BinderProxy.transact(BinderProxy.java:641)"
  ]
}

mainThreadInBinder = true 是核心证据。binderThreadWaitsMain = false 不是推翻结论，它只是说明当前报告没有采集到“本进程 Binder 线程等待主线程”的证据。跨进程阻塞通常还要结合远端进程日志、远端堆栈或 Perfetto 复核。

5

看 pendingQueue，确认主线程已经拖住后续消息

队列里出现多条已经超时等待的消息，例如：

index=1 callbackClass=android.view.View$UnsetPressedState blockedMs=3371
index=2 targetClass=android.view.GestureDetector$GestureHandler blockedMs=3214
index=4 callbackClass=android.graphics.animation.RenderNodeAnimator$$ExternalSyntheticLambda0 blockedMs=2977

这些消息不是根因，而是后果：点击回调中的 Binder 调用没有返回，导致后续按压状态、手势、动画等消息都无法被主线程及时处理。

6

排除 nativePollOnce / Barrier 方向

本次 barrierEvidence 显示：

{
  "repeatedInfinitePollCount": 0,
  "alignedWithPendingBarrier": false,
  "stuckTokens": [],
  "nativePollOnceRecords": []
}

如果是 Sync Barrier 泄漏，通常会看到卡住的 barrier token、nativePollOnce 记录或队列头部屏障证据。本次没有这些证据，所以不要把根因写成 nativePollOnce 空等。

根因链路

用户点击按钮 -> MainActivity.onCreate$lambda$3() -> BinderCrossProcessBlockScenario.run() -> RemoteBlockingServiceClient.blockRemote() -> IRemoteBlockingService.blockFor() -> BinderProxy.transactNative() 同步等待远端进程返回 -> 主线程 3 秒以上无法继续处理消息 -> SDK 捕获疑似 ANR。

问题层级 定位结果
表象 点击按钮后主线程卡住，后续 UI 消息排队。
直接原因 主线程执行同步 Binder 调用，停在 BinderProxy.transactNative。
业务入口 BinderCrossProcessBlockScenario.run(BinderCrossProcessBlockScenario.kt:54)
高风险调用 RemoteBlockingServiceClient.blockRemote(RemoteBinderClient.kt:118) 调用远端 blockFor()。
需要进一步确认 远端进程为什么长时间不返回：远端方法耗时、远端锁等待、远端主线程阻塞、远端 I/O 或远端线程池耗尽。

修复方案

短期修复

  • 不要在主线程直接调用可能耗时的同步 Binder 方法，例如本次的 blockFor()。
  • 把跨进程调用移动到后台线程，主线程只负责显示 loading、禁用按钮和接收结果。
  • 为远端调用增加超时、取消和失败兜底，避免远端不返回时无限等待。
  • 远端服务方法内部不要执行长耗时逻辑；耗时任务应转移到远端后台线程，并通过回调、广播、Flow、LiveData 或轮询结果通知主进程。

长期治理

  • 梳理所有主线程 Binder 调用点，尤其是点击、生命周期、广播、Service 回调、ContentProvider 查询入口。
  • 对关键 Binder 接口定义耗时预算，例如主线程禁止调用超过 100ms 的接口。
  • 在接口设计上优先使用异步协议：请求立即返回任务 ID，结果通过异步通道返回。
  • 线上排查时同时拉取主进程 JSON、远端进程日志、系统 traces 或 Perfetto，用来确认远端实际卡在哪里。

新人排查口诀

先看归因，再看当前消息，再看栈顶，再找第一个业务栈。

这份 JSON 的关键路径是：BINDER_BLOCK_SUSPECTED -> View$PerformClick -> BinderProxy.transactNative -> BinderCrossProcessBlockScenario.run。所以结论应该写“按钮点击中发起同步跨进程 Binder 调用，远端长时间不返回导致主线程阻塞”。

写出根因结论

本次报告是 Demo App “输入事件当前慢消息”场景触发的疑似 ANR。SDK 在按钮点击消息执行期间采集到现场：mainThread.current.callbackClass 为 View$PerformClick，当前消息已执行 3033ms；主线程栈显示阻塞点为 Thread.sleep，业务入口为 CurrentSlowInputScenario.run(CurrentSlowInputScenario.kt:25)，触发来源为 MainActivity 的点击回调。Pending 队列中后续 UI 消息被阻塞约 3023ms，这是当前消息占住主线程后的结果。Barrier、Binder 和系统调度证据均不构成本次主因。因此根因是：按钮点击事件在主线程执行了阻塞逻辑，导致当前主线程消息执行过慢。

3. 本次问题应该怎么修

这类问题的修复重点不是“调大 ANR 阈值”，也不是“忽略这条报告”，而是把点击事件里的阻塞动作从主线程拿走。主线程只负责接收点击、切换按钮状态、展示加载中和更新结果，真正耗时的等待、计算、I/O 或跨进程调用都要放到后台执行。

3.1 本次 Demo 场景的直接修复

当前写法 问题 推荐改法
onClick -> CurrentSlowInputScenario.run() -> Thread.sleep(6000) 点击回调运行在主线程，Thread.sleep 会直接让主线程停止处理后续 UI、输入和生命周期消息。 点击回调只发起任务；耗时逻辑放到后台线程、协程或线程池；任务完成后再切回主线程更新 UI。 
// 反例：点击回调里直接阻塞主线程
button.setOnClickListener {
    CurrentSlowInputScenario().run()
}

// 推荐方向：主线程只发起任务，耗时动作离开主线程
button.setOnClickListener {
    button.isEnabled = false
    backgroundExecutor.execute {
        runSlowWorkOffMainThread()
        mainHandler.post {
            button.isEnabled = true
            renderResult()
        }
    }
}

如果业务使用 Kotlin 协程，也可以用 Dispatchers.IO 或业务专用调度器承载耗时工作，再用 Dispatchers.Main 更新 UI。关键原则只有一个：点击回调本身必须很快返回。

3.2 真实业务里要排查哪些代码

问题点 修复方向 验收方式
点击事件直接在主线程执行耗时等待。 不要在主线程调用 Thread.sleep、同步 I/O、同步锁等待、网络请求或大计算。 再次点击按钮后，mainThread.current.wallMs 不应超过 ANR 阈值。
耗时逻辑和 UI 点击回调耦合。 把耗时任务拆到后台线程、协程、线程池或异步任务；主线程只更新 UI 状态。 主线程栈不再出现业务阻塞方法，Pending 队列不再持续堆积。
低置信度报告也需要人工复核。 保留当前消息、采样栈、Pending 队列、排除项一起看。 分析结论必须能引用至少两个 JSON 节点作为证据。

3.3 修复后的验收标准

验收项 通过标准 怎么看
主线程当前消息耗时 mainThread.current.wallMs 不再超过 SDK 疑似 ANR 阈值，点击回调应尽快结束。 重新触发同一操作，检查新 JSON 或确认不再生成同类疑似 ANR 报告。
主线程栈 栈中不再出现业务阻塞方法，例如 Thread.sleep、同步锁等待、同步网络、同步文件读写。 查看 mainThread.stackFrames 和 stackSamples。
Pending 队列 后续 UI 或输入消息不再长时间 blockedMs。 查看 pendingQueue.messages[0].blockedMs，确认没有持续堆积。
归因结果 不再出现同一入口的 CURRENT_MESSAGE_SLOW，或者归因转为其他真实问题后继续分析。 查看 attribution.primary、evidence 和主线程栈。

3.4 防止同类问题再次出现

  • 点击、滑动、输入、生命周期回调中不做同步等待；必须异步化。
  • 耗时任务统一经过后台执行入口，避免业务代码随手在主线程执行阻塞逻辑。
  • 对高频入口加埋点或日志，记录点击到结果展示的耗时，提前发现慢操作。
  • 上线后按 attribution.primary、业务栈顶方法和页面聚类，优先治理重复出现的入口。
  • 评审代码时重点搜索 Thread.sleep、CountDownLatch.await、Future.get、同步 I/O、主线程锁等待等高风险写法。

ANR 根因分析报告：Sync Barrier 泄漏 / nativePollOnce

分析对象：bd78b591-6e61-48d2-b8a5-568114f1c7a4.json。这份报告的重点是：系统栈看到 nativePollOnce 时，不要停在“主线程空等”这个表象，要继续看 Pending 队列和 Barrier token 证据，找到是谁发布了没有移除的同步屏障。

最终结论

根因主线程消息队列头部存在未移除的 Sync Barrier，导致后面的同步消息无法执行。主线程因此停在 MessageQueue.nativePollOnce 等待异步消息，看起来像“空闲等待”，实际是同步消息被 Barrier 拦住。

Barrier token 是 20，发布入口定位到 SyncBarrierLeakScenario.run(SyncBarrierLeakScenario.kt:55)，触发来源是 MainActivity.runSyncBarrierLeak(MainActivity.kt:62)。

SDK 归因attribution.primary = SYNC_BARRIER_STUCK，置信度 HIGH。

栈顶表象主线程栈顶是 android.os.MessageQueue.nativePollOnce。

队列根因pendingQueue.messages[0].isBarrierLike = true，并且 targetClass = null。

2. 读 SDK 摘要，拿到初始方向

先看 attribution，它相当于这份 JSON 的“先读结论”。本次 SDK 已经直接给出了高置信度方向：

{

    "barrier stuck token=20 alive=3272ms",

新人看到这里可以先记住一句话：SYNC_BARRIER_STUCK 不是说 nativePollOnce 自己有 bug，而是说主线程队列里有一个同步屏障没有被移除，导致同步消息无法继续执行。

3. 看主线程栈，识别 nativePollOnce 表象

主线程栈显示主线程停在 MessageQueue.next() 里，最上层是 nativePollOnce：

android.os.MessageQueue.nativePollOnce(Native Method)
android.os.MessageQueue.next(MessageQueue.java:339)
android.os.Looper.loopOnce(Looper.java:179)
android.os.Looper.loop(Looper.java:344)
android.app.ActivityThread.main(ActivityThread.java:8205)

如果只看这段栈，很容易误判成“主线程没事，只是在等消息”。但 ANR 现场不能只看栈顶。对 nativePollOnce 来说，下一步必须看消息队列：主线程为什么没有拿到可执行消息？是队列真的没消息，还是有消息被 Sync Barrier 挡住了？

本次 mainThread.current = null，说明采集时主线程没有正在执行的业务消息，而是在从消息队列取下一条消息。这个现象正好符合 Sync Barrier 泄漏：队列里有同步消息，但同步消息被队头 Barrier 拦住，主线程只能继续等待异步消息。

4. 看 Pending 队列，确认队头是不是 Barrier

本次根因判断的第一处硬证据在 pendingQueue.messages[0]。队头消息长这样：

targetClass null Android 的 Sync Barrier 没有 target，这是识别 Barrier 的关键特征之一。

6. 看被挡住的同步消息，确认影响范围

队头 Barrier 后面堆着多条同步消息，第一条同步消息已经等待 3278ms：

这里要注意一个判断点：Sync Barrier 的作用是放行异步消息、拦截同步消息。所以主线程进入 nativePollOnce 并不代表队列空了，而是当前没有可执行的异步消息，队列里的同步消息又被 Barrier 挡住。

7. 用 nativePollOnce 增强证据闭环

barrierEvidence.nativePollOnceRecords 记录了主线程进入无限等待的线索：

{
  "timeoutMillis": -1,
  "isInfiniteWait": true,
  "isInFlight": true,
  "source": "STACK_INFERENCE"
}

timeoutMillis = -1 表示无限等待，isInFlight = true 表示采集时仍处于等待中。再结合 alignedWithPendingBarrier = true，就能形成闭环：主线程在 nativePollOnce 等待，队列头部是未移除 Barrier，后面的同步消息被挡住。

5

最后写排除项

Binder、CPU 忙等、系统调度、低内存和存储不足都没有成为主因，因此根因收敛到 Sync Barrier 泄漏。

可直接用于评审的根因描述：

本次 ANR 是 Sync Barrier 泄漏导致的 nativePollOnce 等待。采集时主线程栈停在 MessageQueue.nativePollOnce，但 Pending 队列并非真正空闲：队头是 targetClass=null、isBarrierLike=true 的同步屏障，token 为 20，已经停留约 3294ms。Barrier 证据中 stuckTokens[0].token=20 与队头消息对齐，且 removeUptimeMs=null，发布栈定位到 SyncBarrierLeakScenario.run(SyncBarrierLeakScenario.kt:55)。Barrier 后方多条同步消息已被阻塞约 3278ms；Binder、CPU 忙等和系统调度证据均不成立。因此根因是业务入口发布 Sync Barrier 后没有及时移除，导致同步消息长期无法被主线程调度。

10. 修复方案

直接修复

  • 检查发布 Sync Barrier 的代码路径，确保每一次 postSyncBarrier 都有匹配的 removeSyncBarrier(token)。
  • 把 token 保存到明确的生命周期对象中，不要只放在局部变量里，避免异常、页面销毁、提前 return 后丢失 token。
  • 用 try/finally 或统一清理方法保证失败路径也能移除 Barrier。

生命周期兜底

  • 页面 onStop、onDestroy 或场景取消时清理未移除的 Barrier。
  • 如果业务用了 Choreographer、动画、绘制调度或自定义 Handler 屏障，要给“发布后超时未移除”加监控日志。
  • 避免在普通业务代码中直接反射调用隐藏 API；确实需要屏障语义时，优先使用系统公开调度能力或封装成受控组件。

修复后验收

  • 再次点击同一入口，不再生成 SYNC_BARRIER_STUCK 报告。
  • pendingQueue.messages[0] 不再出现长时间停留的 isBarrierLike=true 消息。
  • barrierEvidence.stuckTokens 为空，或者 token 有明确 removeUptimeMs。
  • 即使系统栈偶尔仍出现 nativePollOnce，也不能再和 Pending 队头 Barrier 对齐。

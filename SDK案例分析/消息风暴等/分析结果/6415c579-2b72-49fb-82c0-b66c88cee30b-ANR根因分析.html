ANR 根因分析报告：消息风暴

分析对象：6415c579-2b72-49fb-82c0-b66c88cee30b.json。本次报告的重点是确认主线程是否被大量重复消息淹没，并判断这些重复消息是否才是输入响应变慢的真正原因。

最终结论

根因点击“消息风暴”后，Demo 向主线程队列一次性投递了大量相同任务，导致队列中堆积 80 条 MessageStormHandler / StormRunnable 消息。主线程当前点击消息尚未结束，后续输入、动画和 Watchdog 消息被延迟处理，因此触发疑似 ANR。

最关键证据

attribution.primary = MESSAGE_STORM

attribution.evidence 中出现 pending repeated target count=80。

pendingQueue.messages 中同类 MessageStormScenario$MessageStormHandler 和 MessageStormScenario$StormRunnable 共 80 条。

排除项

本次不是 Sync Barrier 泄漏：barrierEvidence.stuckTokens = []。

本次不是 Binder 卡死：binderBlock.suspected = false。

本次没有系统级确认 ANR：systemAnr.isConfirmedAnr = false，适合作为 SDK 疑似 ANR 现场分析。

一步一步定位根因

1

先看报告身份

打开 event 节点，确认本次事件 ID 是 6415c579-2b72-49fb-82c0-b66c88cee30b，事件类型是 SUSPECT_ANR，环境是 debug。

这说明它是 SDK 主动捕获的疑似 ANR 现场，不是系统最终确认后的完整 ANR 结论。

2

再看 SDK 给出的优先方向

打开 attribution 节点，最重要的是这三项：

{
  "primary": "MESSAGE_STORM",
  "confidence": "MEDIUM",
  "evidence": [
    "pending repeated target count=80",
    "top thread RenderThread cpu=1150ms"

}

primary 已经指向消息风暴；confidence = MEDIUM 表示证据足够形成排查结论，但仍建议结合 Pending 队列具体内容验证。

打开 mainThread.current，看到当前消息是一次点击分发：

4

打开 Pending 队列找重复消息

pendingQueue.available = true，说明 SDK 成功抓到了主线程消息队列。队列中总共有 85 条消息，其中同类 Demo 消息占 80 条。

重复消息 数量 含义
MessageStormScenario$MessageStormHandler + MessageStormScenario$StormRunnable 80 同一个场景向主线程连续投递大量同类 Runnable，形成消息风暴。
RenderNodeAnimator 2 动画相关消息也被排在后面等待，但数量少，不是主因。
View$UnsetPressedState 1 按钮按压态恢复消息已经等待约 3.47 秒，说明 UI 响应被拖慢。

这里是根因判断的核心：不是某条系统消息自己特别慢，而是业务侧在一次点击后把大量重复任务塞进主线程队列，导致后续消息排队等待。

5

用 blockedMs 判断影响范围

Pending 队列第一条消息 blockedMs = 3471，同类风暴消息的 blockedMs 也集中在约 3468 ms。

这表示这些消息早该执行，却因为主线程当前消息没有结束而持续等待。用户体感上会表现为点击后界面无响应、按压态迟迟不恢复、动画和后续输入延迟。

根因链路

用户点击“消息风暴”按钮后，点击回调进入 View$PerformClick。该回调触发 Demo 场景向主线程队列投递 80 条 MessageStormScenario$StormRunnable。采集时，当前点击消息已经执行约 3488 ms，队列中已有消息等待约 3471 ms。Pending 队列重复目标数量达到 80，且 Barrier、Binder、系统调度和资源证据均不支持其他主因。因此，本次 ANR 根因是主线程消息风暴。

修复方案

1. 不要无限或批量向主线程投递重复任务

如果某个 UI 更新、状态刷新、埋点 flush 或列表刷新可能被频繁触发，先判断队列中是否已经存在同类任务。已有任务未执行时，不再重复投递。

if (!hasPendingRefresh) {
  hasPendingRefresh = true
  mainHandler.post {
    hasPendingRefresh = false
    refreshUi()
  }
}

2. 对高频触发入口做防抖或节流

点击、搜索输入、滑动监听、传感器回调、生命周期连发等入口，不能每次都直接 Handler.post。需要把短时间内的重复触发合并成一次。

mainHandler.removeCallbacks(refreshRunnable)
mainHandler.postDelayed(refreshRunnable, 300)

3. 把可拆分的重任务移出主线程

主线程只做最终 UI 提交。数据计算、批量遍历、IO、解析、排序、压缩等工作放到后台线程，完成后只投递一次 UI 更新。

4. 页面销毁或场景切换时清理旧消息

Activity / Fragment 销毁、列表数据源切换、任务取消时，要清理已经无效的 callbacks 或 messages，避免旧任务继续排队占用主线程。

override fun onDestroy() {
  mainHandler.removeCallbacksAndMessages(ownerToken)
  super.onDestroy()
}

5. 对真实业务增加队列治理指标

建议在 SDK 或业务侧补充同类消息数量、单入口投递次数、去重命中率、队列清理次数等指标。线上看到 MESSAGE_STORM 后，可以快速定位是哪一个业务入口在重复投递。

复查清单

  • 再次复现后，attribution.primary 不应继续稳定出现 MESSAGE_STORM。
  • pendingQueue.messages 中同类业务 Handler / Runnable 数量应明显下降，最好保持个位数。
  • mainThread.current.wallMs 应低于输入事件疑似 ANR 阈值。
  • barrierEvidence 和 binderBlock 继续无异常，避免修复过程中引入新的阻塞类型。
  • 真实业务修复后，要确认 UI 状态仍然正确，不能因为去重或防抖漏掉最后一次用户操作。

可以直接写进问题单的结论

根因：点击“消息风暴”后，业务代码向主线程队列重复投递 80 条 MessageStormScenario$StormRunnable，导致主线程消息队列拥塞。采集时当前 View$PerformClick 已执行约 3488 ms，队列中消息最长等待约 3471 ms。Barrier、Binder、系统调度和资源证据均未显示为主因，因此本次 ANR 归因为主线程消息风暴。

修复建议：对该入口的 Handler.post / sendMessage 增加去重、防抖或节流；页面销毁或任务取消时清理无效 callbacks；重计算或批处理移到后台线程，主线程只保留一次最终 UI 更新。

ANR 根因分析报告：线程池耗尽 + 主线程等待

分析对象：5596b983-ec89-4fbd-8eea-460155efc13b.json。本次最可信的根因是：按钮点击回调在主线程中等待线程池排队任务结果，而线程池 worker 已被长任务占满，导致主线程停在 FutureTask.get()，当前点击消息超过 3 秒没有结束。

结论

根因主线程同步等待线程池任务结果。

代码入口是 ThreadPoolExhaustionWaitScenario.run()，阻塞点在 ThreadPoolExhaustionWorkload.exhaustPoolAndWait() 内部调用 FutureTask.get()。

关键证据

attribution.primary = CURRENT_MESSAGE_SLOW

mainThread.current.wallMs = 3352ms

mainThread.current.cpuMs = 1ms

耗时很长但 CPU 很低，说明主线程更像是在等待，而不是 CPU 忙等。

排除项

barrierEvidence.stuckTokens = []，不是 Sync Barrier 泄漏主因。

binderBlock.suspected = false，不是 Binder 跨进程阻塞主因。

链路阶段 证据 说明

主线程等待结果 FutureTask.get、LockSupport.parkNanos、Unsafe.park 主线程同步等待排队任务结果，导致 Looper 无法处理后续消息。
ANR 现场表现 wallMs = 3352ms，pendingQueue 后续消息 blocked 超过 3 秒 当前点击消息长时间未结束，UI 状态恢复、手势、动画和 Watchdog 消息都被拖住。

修复方案

直接修复

  • 不要在主线程调用 Future.get()、CountDownLatch.await()、CompletableFuture.get()、runBlocking 等同步等待 API。
  • 点击回调只负责发起任务和更新轻量 UI 状态，例如禁用按钮、展示加载中。
  • 耗时任务放到后台线程、协程 Dispatchers.IO / Dispatchers.Default、业务线程池或 WorkManager。
  • 任务完成后通过回调、LiveData、Flow、StateFlow 或主线程 Handler 回到 UI 层更新结果。

线程池治理

  • 不要让“提交任务”和“等待任务结果”的代码使用同一个容易被占满的小线程池。
  • 为关键链路配置超时、取消和降级路径，避免主流程无限等待后台资源。
  • 按业务类型拆分线程池：CPU 计算、磁盘 I/O、网络、低优先级预加载不要全部挤在同一个池里。
  • 监控线程池队列长度、活跃线程数、拒绝次数和任务等待时间，提前发现耗尽风险。

修复后验收

验收项 期望结果
再次点击同一按钮 不再生成同入口的 CURRENT_MESSAGE_SLOW 报告。
检查主线程栈 不再出现 FutureTask.get、LockSupport.parkNanos、Unsafe.park 等等待栈。
检查 mainThread.current wallMs 回落到阈值以内，点击消息能快速结束。
检查 pendingQueue 后续 UI、手势、动画和 Watchdog 消息不再长时间 blocked。

可直接用于评审的结论

本次 ANR 是“线程池耗尽 + 主线程等待”导致的当前消息慢。SDK 报告显示 attribution.primary = CURRENT_MESSAGE_SLOW，当前主线程消息为 View$PerformClick，已执行 3352ms 且尚未结束；同时 cpuMs = 1ms，说明主线程不是 CPU 忙等，而是在等待。主线程栈明确停在 FutureTask.get，调用链回溯到 ThreadPoolExhaustionWorkload.exhaustPoolAndWait(ThreadPoolExhaustionWorkload.kt:58) 和 ThreadPoolExhaustionWaitScenario.run(ThreadPoolExhaustionWaitScenario.kt:34)。Pending 队列中后续 UI、手势、动画和 Watchdog 消息已经被阻塞约 3 秒；Barrier 与 Binder 专项证据均不成立。因此根因是点击回调在主线程同步等待线程池排队任务结果，而线程池 worker 被长任务占满，导致主线程无法继续处理消息。

一句话修复方向

把线程池任务等待从主线程移除，改为异步回调或状态通知；同时治理线程池容量、队列、超时和降级，避免关键交互依赖已经耗尽的线程池。

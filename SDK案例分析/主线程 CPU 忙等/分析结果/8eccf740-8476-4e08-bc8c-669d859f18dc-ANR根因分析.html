ANR 根因分析报告：主线程 CPU 忙等

分析对象：8eccf740-8476-4e08-bc8c-669d859f18dc.json。这份报告的重点是区分“主线程在等待”和“主线程一直在计算”。本次证据显示，点击回调进入主线程 busy loop，导致当前消息超过疑似 ANR 阈值。

最终结论

根因按钮点击消息在主线程持续执行 CPU 密集型循环，导致主线程无法及时处理后续输入、绘制和 Watchdog 消息。

具体业务入口是 MainThreadCpuBusyScenario.run(MainThreadCpuBusyScenario.kt:29)，直接耗 CPU 的位置是 DefaultCpuBusyAction.burn(MainThreadCpuBusyScenario.kt:37)。

SDK 归因attribution.primary = CURRENT_MESSAGE_SLOW

当前消息耗时wallMs = 3086ms，超过 3000ms 阈值。

CPU 忙等特征cpuMs = 3070ms，几乎等于墙钟耗时。

2. 先读 SDK 摘要，拿到初始方向

SDK 给出的主归因是 CURRENT_MESSAGE_SLOW，意思是：采集时主线程正在处理一条消息，这条消息已经执行太久。

{
  "primary": "CURRENT_MESSAGE_SLOW",
  "confidence": "MEDIUM",
  "evidence": [
    "current message wall=3086ms cpu=3070ms",
    "top thread beanrmonitoring cpu=3810ms"
  ]
}

这里不能只停在 CURRENT_MESSAGE_SLOW。它只说明“当前消息慢”，还要继续判断慢的原因是等待、锁、I/O、Binder，还是 CPU 忙等。本次 wallMs 和 cpuMs 非常接近，这是 CPU 忙等的强信号。

1

先写现象

这是一份 SUSPECT_ANR 报告，SDK 归因为 CURRENT_MESSAGE_SLOW，表示主线程正在执行的点击消息超过阈值。

2

再写证据

当前消息 wallMs=3086ms，cpuMs=3070ms，主线程 CPU 排名第一，说明主线程一直在消耗 CPU。

3

然后写业务入口

主线程栈命中 MainThreadCpuBusyScenario.run 和 DefaultCpuBusyAction.burn，调用链来自 MainActivity 的点击回调。

4

最后写排除项

Barrier、Binder、系统调度、低内存和存储不足都没有成为主因，因此根因收敛到主线程 CPU 密集型逻辑。

可直接用于评审的根因描述：

本次 ANR 是“主线程 CPU 忙等”导致的当前消息慢。用户点击按钮后，主线程进入 MainThreadCpuBusyScenario.run，并在 DefaultCpuBusyAction.burn 中持续计算，导致当前点击消息执行 3086ms。同时 cpuMs=3070ms，主线程位于 CPU 排名第一，说明主线程不是等待阻塞，而是在持续消耗 CPU。Pending 队列中后续输入和 Watchdog 消息已排队超过 3 秒；Barrier 和 Binder 证据均不成立，因此根因是点击回调中存在主线程 CPU 密集型逻辑。

9. 修复方案

立即修复

  • 不要在点击回调、生命周期回调、绘制回调中执行长时间计算。
  • 把 DefaultCpuBusyAction.burn 这类 CPU 密集型逻辑移到后台线程、协程 Dispatchers.Default 或业务线程池。
  • 主线程只负责发起任务、展示 loading、接收结果并刷新 UI。

拆分和降级

  • 如果计算必须分多步完成，把大任务拆成小批次，每批结束后让出主线程。
  • 给耗时任务增加取消能力，页面退出或重复点击时取消旧任务。
  • 对按钮做防重复点击，避免同类 CPU 任务被连续触发。

上线前验证

  • 修复后重新点击“当前消息忙等”对应业务入口，确认不再生成 CURRENT_MESSAGE_SLOW 报告。
  • 确认主线程 wallMs 不再超过阈值，cpuMs 不再接近墙钟耗时。
  • 确认 threadCpu.topThreads 中主线程不再长期排名第一。

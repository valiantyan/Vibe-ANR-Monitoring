ANR 根因分析报告：IO / 数据库 / 文件阻塞

分析对象：2eab043f-88b2-4b0e-89c4-5f237f0f4585.json。本次最可信的根因是主线程在按钮点击回调中执行文件和数据库阻塞工作，尤其是在事务内批量插入 payload 行并伴随大量进程写入，导致当前主线程消息超过 3 秒未结束。

结论

根因点击回调在主线程执行数据库事务和文件/IO 工作。

业务入口是 IoDatabaseFileBlockScenario.run()，核心阻塞链路进入 FileAndDatabaseBlockingWorkload.runIoDatabaseFileWorkload()、runBlockingDatabaseTransaction() 和 insertPayloadRow()。

用户点击按钮 -> MainActivity.onCreate$lambda$8() -> IoDatabaseFileBlockScenario.run() -> FileAndDatabaseBlockingWorkload.runIoDatabaseFileWorkload() -> runBlockingDatabaseTransaction() -> insertRowsInTransaction() -> insertPayloadRow() 在主线程持续执行数据库插入和 payload 构造，同时进程产生大量写入 -> 当前点击消息超过 3 秒未完成 -> 后续 UI/动画/Watchdog 消息排队 -> SDK 捕获疑似 ANR。

问题层级 定位结果
表象 点击按钮后主线程卡住，当前 View$PerformClick 消息未结束，后续消息开始排队。
直接原因 主线程执行耗时数据库事务和文件/IO 写入相关工作。
业务入口 IoDatabaseFileBlockScenario.run(IoDatabaseFileBlockScenario.kt:45)
核心阻塞点 FileAndDatabaseBlockingWorkload.insertPayloadRow(FileAndDatabaseBlockingWorkload.kt:113)，上游为 insertRowsInTransaction(FileAndDatabaseBlockingWorkload.kt:93) 和 runBlockingDatabaseTransaction(FileAndDatabaseBlockingWorkload.kt:81)。
证据强度 SDK 置信度为 MEDIUM。虽然不是 HIGH，但主线程当前消息、主线程栈、采样栈和进程写入量互相印证，足以把修复方向定位到主线程 IO/数据库工作。

修复建议

短期修复

  • 不要在主线程点击回调中直接执行数据库批量插入、文件写入或大 payload 构造。
  • 把 runIoDatabaseFileWorkload 这类重工作移动到后台线程，例如协程 Dispatchers.IO、专用线程池或 WorkManager。
  • 主线程只负责触发任务、更新 loading/按钮状态和接收结果，避免一次点击消息长时间占用 Looper。
  • 批量数据库写入要控制单批大小，必要时分批提交，并避免在事务内做不必要的字符串格式化或大对象构造。

长期治理

  • 为主线程数据库、文件、SharedPreferences、ContentProvider 查询等调用建立静态扫描或运行时 StrictMode 监控。
  • 给 Demo 和 SDK 样例补充“主线程禁止阻塞 I/O”的说明，让使用者能把本案例和真实业务点击卡顿对应起来。
  • 线上排查同类问题时，同时关注 mainThread.current、业务栈、processIo.writeBytes、数据库慢查询日志和系统 traces。

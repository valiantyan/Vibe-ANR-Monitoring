ANR 根因分析报告：Service 超时

分析对象：f069605f-74ac-4edf-9a19-3124b6745925.json。这份报告的重点是：不能只看 attribution.primary，要用主线程当前消息和主线程栈复核 SDK 摘要。本次最可信的业务根因是 ServiceTimeoutService.onStartCommand() 在主线程执行阻塞。

最终结论

根因应用内显式 Service 被启动后，ServiceTimeoutService.onStartCommand() 在主线程执行阻塞逻辑，内部调用 Thread.sleep()，导致 ActivityThread 的 Service 分发消息长时间无法结束。

直接阻塞位置是 ServiceTimeoutBlocker$ThreadSleepBlockingAction.block(ServiceTimeoutBlocker.kt:27)，业务入口是 ServiceTimeoutService.onStartCommand(ServiceTimeoutService.kt:33)。

最关键证据

当前消息mainThread.current.targetClass = android.app.ActivityThread$H，what = 115，并且 isCriticalComponent = true。

耗时mainThread.current.wallMs = 3375ms，超过 Demo 的疑似 ANR 阈值。

主线程栈栈中明确出现 ServiceTimeoutService.onStartCommand、ServiceTimeoutBlocker.block 和 Thread.sleep。

排除项

非 BinderbinderBlock.suspected = false

非 CPU 忙等mainThread.current.cpuMs = 0，主线程更像睡眠或等待，不是持续计算。

非已确认系统 ANRsystemAnr.isConfirmedAnr = false，这是一份 SDK 提前抓到的疑似 ANR 现场。

9. 修复方案

直接修复

  • 不要在 Service.onStartCommand()、onCreate()、onBind() 等 Service 生命周期回调里执行 Thread.sleep()、同步网络请求、同步磁盘 I/O、长锁等待或大计算。
  • onStartCommand() 只做参数校验、任务入队和快速返回。耗时任务放到后台线程、协程 Dispatchers.IO、Executor、WorkManager 或 JobScheduler。
  • 如果必须前台长期运行，使用前台服务并尽快调用 startForeground()，但这不代表可以在主线程做耗时任务。

代码层检查清单

检查点 应该怎么改
onStartCommand() 里有睡眠、等待、同步 I/O 移动到后台执行，只保留轻量调度逻辑。
主线程等待后台任务结果，例如 Future.get()、CountDownLatch.await() 改成异步回调或状态流转，不要阻塞主线程等结果。
Service 和 Activity/Receiver 之间有锁互等 缩小锁范围，避免主线程进入长时间同步块。
任务需要系统保障执行 优先评估 WorkManager 或 JobScheduler，不要靠 Service 主线程硬扛。

修复后如何验证

  • 再次点击 Demo 的“Service 超时”或业务中的对应入口，确认不会继续生成同类疑似 ANR 报告。
  • 如果仍生成 JSON，重点看 mainThread.stackFrames 是否还出现 ServiceTimeoutService.onStartCommand 或业务 Service 的生命周期方法。
  • 确认 mainThread.current.wallMs 不再持续超过阈值，Pending 队列中组件关键消息不再长时间堆积。

10. 可以直接写进问题单的结论

根因：Service 启动后，主线程在 ServiceTimeoutService.onStartCommand() 中执行阻塞逻辑，调用链为 ServiceTimeoutService.onStartCommand -> ServiceTimeoutBlocker.block -> Thread.sleep。该 Service 生命周期消息由 ActivityThread$H 分发，采集时已执行 3375ms 且尚未结束，导致主线程无法继续处理后续消息。

排除：虽然 Pending 队头存在疑似 Sync Barrier，但 barrierEvidence.stuckTokens=[]、alignedWithPendingBarrier=false、nativePollOnceRecords=[]，且主线程栈不在 nativePollOnce，所以 Barrier 不是本次主因。binderBlock.suspected=false，主线程 cpuMs=0，也不支持 Binder 阻塞或 CPU 忙等。

修复：将 Service 生命周期回调中的耗时逻辑移到后台线程或任务调度框架中，onStartCommand() 只做轻量入队并快速返回。

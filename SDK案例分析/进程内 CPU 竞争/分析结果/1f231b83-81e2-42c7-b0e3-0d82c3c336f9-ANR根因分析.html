ANR 根因分析报告：进程内 CPU 竞争

分析对象：1f231b83-81e2-42c7-b0e3-0d82c3c336f9.json。本次最可信的根因是：同一进程内多个后台 DemoCpuContender-* 线程持续消耗 CPU，主线程当前 Activity 启动消息 wall time 超过 3 秒，但主线程 CPU 只有 160ms，表现为主线程被后台 CPU 竞争拖慢，而不是主线程自身 busy loop。

结论

根因进程内后台 CPU 密集线程抢占调度资源，导致主线程当前消息响应变慢。

业务入口是 ProcessCpuContentionScenario.run()，工作负载入口是 DefaultProcessCpuContentionWorkload.createContentionAndWaitOnMainThread()。

关键证据

attribution.primary = CURRENT_MESSAGE_SLOW

mainThread.current.wallMs = 3021ms

mainThread.current.cpuMs = 160ms

threadCpu.topThreads[0] = DemoCpuContende / 2570ms

排除项

barrierEvidence.stuckTokens = []，不是 Sync Barrier 泄漏。

binderBlock.suspected = false，不是 Binder 或跨进程阻塞。

mainThread.current.cpuMs = 160ms，不符合“主线程自己持续计算”的典型特征。

JSON 节点目录

节点 本次重点值 怎么理解
event eventType = SUSPECT_ANR，eventId = 1f231b83... 这是 SDK 在 debug 环境捕获到的疑似 ANR 现场，用 eventId 和 logcat 的 report id 对齐。
attribution primary = CURRENT_MESSAGE_SLOW，confidence = LOW SDK 摘要先判断“当前消息慢”。置信度低时要继续看主线程栈和线程 CPU 排名。

一步一步定位根因

1

SUSPECT_ANR 表示 SDK 抓到的是疑似 ANR 现场。它不一定已经被系统弹框确认，但可以用于分析主线程为什么在当时超过阈值。

2

6

区分“主线程 CPU 忙等”和“进程内 CPU 竞争”

判断项 本次值 说明
主线程 wall time 3021ms 当前消息确实慢。
主线程 CPU 160ms 主线程自身没有持续占满 CPU。
后台线程 CPU 2360ms - 2570ms 后台竞争线程 CPU 明显高。

所以本次不能写成“主线程忙等”。更准确的说法是：后台 CPU 密集线程抢占进程内调度资源，主线程当前消息被拖慢。

10

写出最终根因

本次 ANR 现场由进程内 CPU 竞争场景触发。当前 ActivityThread 主消息执行超过 3 秒，但主线程 CPU 只有 160ms；同时多个 DemoCpuContender-* 后台线程位于 CPU 排名前列，单线程 CPU 约 2.3 到 2.5 秒。Barrier 和 Binder 证据均不成立。因此根因应写为：同一进程内后台 CPU 密集任务持续抢占调度资源，导致主线程当前消息响应延迟。

修复方案

问题点 修复建议 验收方式
后台 CPU 任务并发过高 限制 CPU 密集任务并发数，通常不要超过可用核心数；前台交互期间进一步降并发。 再次采集 JSON，threadCpu.topThreads 不应持续被同类业务线程占据前列。
CPU 密集任务和 UI 抢调度 将计算任务放入专用低优先级线程池，避免和主线程、RenderThread、Binder 线程争抢资源。 验证主线程 wallMs 下降，且用户操作期间不再触发疑似 ANR。
批量计算不可中断 把大任务拆成小批次，批次之间检查取消标记和业务优先级，必要时让出调度窗口。 压测期间观察单个任务耗时、取消成功率和 CPU 峰值。
线上定位困难 给 CPU 密集任务设置清晰线程名和任务名，记录任务来源、开始时间、耗时和取消原因。 线上 JSON 中能直接看到业务线程名，而不是匿名线程或通用线程池名。

最终结论模板

本次 ANR 是进程内 CPU 竞争导致的当前消息响应延迟。SDK 报告显示当前主线程消息 wallMs=3021ms，但主线程 cpuMs=160ms，说明主线程自身不是持续计算；同时 threadCpu.topThreads 前列均为 Demo CPU 竞争线程，单线程 CPU 约 2360ms - 2570ms。主线程栈能回溯到 ProcessCpuContentionScenario.run 和 DefaultProcessCpuContentionWorkload.createContentionAndWaitOnMainThread。Barrier 与 Binder 证据均不成立，因此根因是同一进程内后台 CPU 密集任务持续抢占调度资源，导致主线程当前消息超过疑似 ANR 阈值。

ANR 根因分析报告：主线程 SQLite / 文件数据库阻塞

分析对象：a01e2e96-9919-488f-8224-b2709b8f8c1a.json。虽然目录名是 LooperPrinter 竞争验收，但这份 JSON 的主线程栈明确指向 IoDatabaseFileBlockScenario 和 FileAndDatabaseBlockingWorkload：主线程正在结束 SQLite 事务，当前消息执行超过 3 秒，且主线程 CPU 较高。因此本次根因应按主线程同步数据库事务 / 文件数据库阻塞分析。

结论

根因主线程执行同步 SQLite 事务和文件数据库工作负载，导致 ActivityThread 当前消息超过疑似 ANR 阈值。

直接入口是 IoDatabaseFileBlockScenario.run()，核心耗时栈在 FileAndDatabaseBlockingWorkload.insertRowsInTransaction() 和 SQLiteDatabase.endTransaction()。

9

确认 LooperPrinter 竞争不是本次 JSON 的主因

这份报告目录叫 LooperPrinter 竞争验收，但 JSON 中没有看到 Looper Printer 竞争、锁等待或打印器相关栈。根因分析应以 JSON 的主线程栈为准：SQLiteConnection.nativeExecute、SQLiteDatabase.endTransaction 和 FileAndDatabaseBlockingWorkload 才是本次能落地修复的证据。

10

写出最终根因

本次 ANR 是主线程同步执行数据库 / 文件工作负载导致的当前消息慢。ActivityThread 当前消息 wallMs=3381ms，cpuMs=2439ms；主线程栈停在 SQLite 事务结束阶段，并能回溯到 IoDatabaseFileBlockScenario.run。Pending 队列中多个关键消息已经阻塞超过 3 秒。Barrier 和 Binder 证据均不成立，因此根因应写为“主线程执行同步 SQLite 事务和文件数据库操作，导致关键主线程消息长时间未返回”。

修复方案

问题点 修复建议 验收方式
主线程执行 SQLite 事务 把批量插入、事务提交、索引维护等数据库操作移到后台线程，主线程只做状态更新。 再次采集 JSON，主线程栈不应再出现业务数据库事务调用。
单次事务过大 拆分批次，每批设置数量上限；批次之间允许取消、降级或让出调度窗口。 单条主线程消息 wallMs 低于 ANR 阈值，Pending 队列不再积压关键组件消息。
文件 / 数据库写入集中爆发 减少主线程路径上的同步写入；必要写入使用异步队列和背压，避免短时间大量刷盘。 environmentSnapshot.processIo.writeBytes 在前台交互窗口明显下降。
线上问题难定位 给数据库操作记录表名、批次大小、事务耗时、线程名和调用来源，并和 ANR eventId 对齐。 线上报告能直接从业务日志定位到具体表和具体批次。

最终结论模板

本次 ANR 是主线程同步数据库 / 文件操作导致的当前消息慢。SDK 报告显示 attribution.primary=CURRENT_MESSAGE_SLOW，当前 ActivityThread 消息 wallMs=3381ms、cpuMs=2439ms；主线程栈停在 SQLiteConnection.nativeExecute / SQLiteDatabase.endTransaction，并能回溯到 FileAndDatabaseBlockingWorkload.insertRowsInTransaction 与 IoDatabaseFileBlockScenario.run。Pending 队列中关键组件消息已阻塞 3 秒以上。Sync Barrier 和 Binder 证据均不成立。因此根因是业务在主线程执行同步 SQLite 事务和文件数据库写入，导致主线程关键消息无法及时返回。

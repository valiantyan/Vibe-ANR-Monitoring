ANR 根因分析报告：BroadcastReceiver 超时

分析对象：06b3b0f2-0bca-4dca-bf96-de61d901a95f.json。这份报告的重点是：不要只看 attribution.primary，要把 SDK 摘要、主线程当前消息、主线程栈和专项证据放在一起核对。本次最可信的业务根因是 BroadcastTimeoutReceiver.onReceive() 在主线程阻塞。

最终结论

根因应用内显式广播到达后，BroadcastTimeoutReceiver.onReceive() 在主线程执行阻塞逻辑，内部调用 Thread.sleep()，导致 ActivityThread 的 Receiver 分发消息长时间无法结束。

直接阻塞位置是 BroadcastTimeoutBlocker$ThreadSleepBlockingAction.block(BroadcastTimeoutBlocker.kt:27)，业务入口是 BroadcastTimeoutReceiver.onReceive(BroadcastTimeoutReceiver.kt:23)。

最关键证据

当前消息mainThread.current.targetClass = android.app.ActivityThread$H，what = 113，并且 isCriticalComponent = true。

耗时mainThread.current.wallMs = 3316ms，超过 Demo 的疑似 ANR 阈值。

主线程栈栈中明确出现 BroadcastTimeoutReceiver.onReceive 和 Thread.sleep。

排除项

非 BinderbinderBlock.suspected = false

非 CPU 忙等mainThread.current.cpuMs = 0，阻塞更像等待或睡眠，不是持续计算。

非已确认系统 ANRsystemAnr.isConfirmedAnr = false，这是一份 SDK 提前抓到的疑似 ANR 现场。

3. 看 SDK 摘要，但不要停在摘要

本次有一个摘要误导点

attribution.primary = SYNC_BARRIER_STUCK，置信度是 HIGH。如果只看这一行，容易写成“Sync Barrier 泄漏导致 ANR”。但这份报告里的主线程栈明确在 Receiver 的 onReceive() 中睡眠，Barrier 专项证据又不完整，所以不能把 Barrier 当成本次业务根因。

摘要字段 值 复核结论
attribution.primary SYNC_BARRIER_STUCK 需要下钻复核。它来自 Pending 队头像 Barrier，但还不能覆盖主线程栈证据。
attribution.evidence[0] pending queue head is Sync Barrier 说明队头消息形态像 Barrier，是一个需要检查的信号。
attribution.evidence[2] nativePollOnce infiniteWaitCount=0 alignedWithPendingBarrier=false 这里反而说明 nativePollOnce 与 Barrier 没有对齐，削弱 Barrier 根因。
attribution.suggestions 检查 Sync Barrier 配对 本次不应作为第一修复方向；应优先修复 Receiver 主线程阻塞。

第三步结论：SDK 摘要给了一个需要复核的方向，但主线程栈和 Barrier 细节不支持把 Barrier 写成最终根因。本次分析必须以主线程现场为准。

主线程栈是本次最有价值的证据。读栈时从上往下看：顶部是当前卡住的位置，往下能看到是谁调用到这里。

Thread.sleep 主线程当前正在睡眠等待，直接导致消息无法结束。

第五步结论：根因已经可以落到代码位置。阻塞不是发生在普通点击回调，也不是后台线程，而是发生在 BroadcastReceiver 的 onReceive() 主线程路径里。

6. 复核 Pending 队列和 Barrier 干扰项

这份报告里 pendingQueue.messages[0] 看起来像 Sync Barrier，所以必须单独解释，否则新人很容易被 attribution.primary 带偏。

字段 值 怎么理解
pendingQueue.messages[0].isBarrierLike true 队头消息形态像 Barrier，这是 SDK 摘要判断 Barrier 的来源。
pendingQueue.messages[0].blockedMs 3312ms 它已经过期 3.3 秒未执行，但这和当前 Receiver 阻塞时间高度重叠。
barrierEvidence.stuckTokens [] 没有观察到未移除的 Barrier token。
barrierEvidence.alignedWithPendingBarrier false Barrier 证据没有和 Pending 队头对齐。
barrierEvidence.nativePollOnceRecords [] 没有 nativePollOnce 无限等待记录。
判断口径：只有“主线程栈停在 nativePollOnce + 队头 Barrier + stuck token 未移除 + Barrier 与 nativePollOnce 对齐”同时成立时，才适合把 Sync Barrier 写成主因。本次主线程栈不在 nativePollOnce，而是在 Thread.sleep -> BroadcastTimeoutReceiver.onReceive，所以 Barrier 只能作为现场中的干扰信号。

9. 修复建议

问题点 修复方向 落地建议
onReceive() 主线程睡眠或耗时 不要在 Receiver 主线程执行长耗时逻辑。 把耗时工作转到后台任务；onReceive() 只做参数校验、轻量分发和快速返回。
广播需要继续异步处理 使用 goAsync() 或调度到 WorkManager/JobScheduler。 使用 goAsync() 时必须在后台任务完成后调用 PendingResult.finish()，并设置超时保护。
Receiver 中存在 I/O、网络、锁等待 移动到业务后台层，并增加取消、超时和降级。 禁止在 onReceive() 中直接做数据库慢查询、网络请求、大文件读写、长锁等待。
误报为 Barrier 优化分析口径或 SDK 规则。 当 mainThread.current 是关键组件消息且主线程栈有明确业务阻塞时，应优先输出当前消息慢或组件阻塞；Barrier 只作为辅因或待复核证据。

10. 可以直接放进问题单的结论

本次报告 06b3b0f2-0bca-4dca-bf96-de61d901a95f 为 SDK 疑似 ANR 报告，系统尚未确认 ANR。主线程当前正在处理 android.app.ActivityThread$H 的关键组件消息，已执行 3316ms。主线程栈显示执行路径为 ActivityThread.handleReceiver -> BroadcastTimeoutReceiver.onReceive -> BroadcastTimeoutBlocker.block -> Thread.sleep，因此业务根因是 BroadcastReceiver 的 onReceive() 在主线程执行阻塞逻辑，导致 Receiver 分发消息无法及时完成。虽然 Pending 队头存在 Barrier-like 消息，SDK 摘要给出 SYNC_BARRIER_STUCK，但本次 barrierEvidence.stuckTokens=[]、alignedWithPendingBarrier=false、nativePollOnceRecords=[]，且主线程并未停在 nativePollOnce，所以不能把 Sync Barrier 作为最终根因。

11. 后续验证建议

  • 修复后重新点击“BroadcastReceiver 超时”按钮，确认不再生成新的疑似 ANR JSON。
  • 如果业务确实需要异步处理广播，验证 goAsync() 的 finish() 一定会被调用。
  • 保留本次 JSON 作为 SDK 规则优化样本：当主线程当前消息和栈已经明确指向 Receiver 阻塞时，归因摘要不应优先给出 SYNC_BARRIER_STUCK。
  • 如果要验证系统确认广播 ANR，可延长阻塞时间并观察 systemAnr.isConfirmedAnr、anrType 是否变成 BROADCAST_FOREGROUND 或 BROADCAST_BACKGROUND。

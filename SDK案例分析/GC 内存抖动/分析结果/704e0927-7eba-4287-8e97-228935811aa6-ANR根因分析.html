ANR 根因分析报告：GC / 内存抖动

分析对象：704e0927-7eba-4287-8e97-228935811aa6.json。本次最可信的根因是：按钮点击回调在主线程执行大量对象分配和保留队列维护，触发明显 GC 压力，导致当前点击消息超过 3 秒没有结束，后续 UI、手势、动画和 Watchdog 消息持续排队。

结论

根因主线程执行 GC / 内存抖动场景，当前点击消息被对象分配与保留队列处理拖慢。

代码入口是 GcMemoryChurnScenario.run()，主要耗时点在 GcMemoryChurnWorkload.churnMemoryOnMainThread()。

关键证据

attribution.primary = CURRENT_MESSAGE_SLOW

mainThread.current.wallMs = 3497ms

mainThread.current.cpuMs = 836ms

threadCpu.topThreads[0] = HeapTaskDaemon / 3250ms

排除项

barrierEvidence.stuckTokens = []，不是 Sync Barrier 泄漏主因。

binderBlock.suspected = false，不是 Binder 或跨进程阻塞主因。

processIo.readBytes = 0，不是明显磁盘读阻塞主因。

一步一步定位根因

1

先确认报告类型

{
  "eventId": "704e0927-7eba-4287-8e97-228935811aa6",
  "eventType": "SUSPECT_ANR",
  "appId": "vibe-anr-demo",
  "environment": "debug"
}

SUSPECT_ANR 表示 SDK 在系统正式 ANR 前保存了现场。分析时不要因为 systemAnr.isConfirmedAnr = false 就忽略它；这类报告通常更接近卡顿发生的第一现场。

2

看 attribution，拿到第一判断

{
  "primary": "CURRENT_MESSAGE_SLOW",
  "confidence": "LOW",
  "evidence": [
    "current message wall=3497ms cpu=836ms",
    "top thread HeapTaskDaemon cpu=3250ms"
  ]
}

SDK 初步判断是 CURRENT_MESSAGE_SLOW，也就是“当前正在 dispatch 的主线程消息过慢”。这里已经出现两个重要信号：当前消息墙钟耗时超过 3 秒，且 HeapTaskDaemon CPU 很高。

confidence = LOW 的含义是：摘要本身不足以单独定责，需要继续用主线程栈、采样栈、队列和排除项补齐证据链。

3

看 mainThread.current，确认卡住的是哪条消息

{
  "kind": "CURRENT",
  "messageType": "looper_dispatch",
  "targetClass": "android.view.ViewRootImpl$ViewRootHandler",
  "callbackClass": "android.view.View$PerformClick@eff63e",
  "wallMs": 3497,
  "cpuMs": 836,
  "endUptimeMs": null
}

这一步可以得到三个结论：

  • callbackClass = View$PerformClick：问题发生在按钮点击回调内。
  • wallMs = 3497ms：这次点击消息已经执行 3.497 秒，超过 SDK 疑似 ANR 阈值。
  • endUptimeMs = null：采集时这条消息还没执行完，后续消息只能排队。
4

看 mainThread.stackFrames，把问题定位到业务代码

[
  "com.valiantyan.vibeanrmonitoring.scenario.GcMemoryChurnWorkload.churnMemoryOnMainThread(GcMemoryChurnWorkload.kt:27)",
  "com.valiantyan.vibeanrmonitoring.scenario.GcMemoryChurnScenario.run(GcMemoryChurnScenario.kt:37)",
  "com.valiantyan.vibeanrmonitoring.MainActivity.onCreate$lambda$10(MainActivity.kt:106)",
  "android.view.View.performClick(View.java:7498)"
]

这是本次最关键的定位证据。栈从下往上看：用户点击按钮后进入 MainActivity 的点击回调，再进入 GcMemoryChurnScenario.run()，最终停在 GcMemoryChurnWorkload.churnMemoryOnMainThread()。

因此不能把结论写成“系统卡顿”或“nativePollOnce”。这份 JSON 里主线程栈已经明确指向 Demo 自己的 GC / 内存抖动代码。

5

看 stackSamples，确认采样时正在做什么

[
  "java.util.ArrayDeque.removeFirst(ArrayDeque.java:268)",
  "com.valiantyan.vibeanrmonitoring.scenario.GcMemoryChurnWorkload.churnMemoryOnMainThread(GcMemoryChurnWorkload.kt:30)",
  "com.valiantyan.vibeanrmonitoring.scenario.GcMemoryChurnScenario.run(GcMemoryChurnScenario.kt:37)"
]

采样栈显示主线程正在 ArrayDeque.removeFirst 和 churnMemoryOnMainThread 之间运行。结合场景名可以判断：代码在主线程持续分配对象、保留一部分对象，并维护保留队列；这会制造频繁分配和回收压力。

6

看 threadCpu，确认 GC 压力证据

{
  "topThreads": [
    { "threadName": "HeapTaskDaemon", "totalCpuMs": 3250 },
    { "threadName": "beanrmonitoring", "totalCpuMs": 1700 },
    { "threadName": "RenderThread", "totalCpuMs": 1160 }
  ]
}

HeapTaskDaemon 是堆和 GC 相关的运行时线程。本次它的 CPU 时间达到 3250ms，排在第一位，说明报告窗口内 GC / 堆维护活动很重。它不是唯一根因，但它和主线程业务栈共同支持“内存抖动导致当前点击消息变慢”的结论。

7

看 pendingQueue，理解后续消息为什么也慢

[
  { "index": 0, "blockedMs": 3488, "isBarrierLike": true, "targetClass": null },
  { "index": 1, "blockedMs": 3432, "callbackClass": "android.view.View$UnsetPressedState" },
  { "index": 2, "blockedMs": 3264, "targetClass": "android.view.GestureDetector$GestureHandler" },
  { "index": 3, "blockedMs": 3032, "callbackClass": "android.graphics.animation.RenderNodeAnimator$$ExternalSyntheticLambda0" },

这里的排队消息不是主因，而是当前点击回调长时间占住主线程后的结果。View$UnsetPressedState、手势、动画和 Watchdog 都已经过期 3 秒左右，说明 UI 反馈和监控探针都被当前消息挡住了。

队头有一条 isBarrierLike = true 的消息，但专项证据没有对齐为 Barrier 泄漏，因此不能只凭队列头一条疑似 Barrier 就下 Sync Barrier 泄漏结论。

看排除项，避免误判

可能方向 JSON 证据 本次结论

Binder / 跨进程阻塞 binderBlock.suspected = false，mainThreadInBinder = false 排除为主因。

根因链路

用户点击“GC / 内存抖动”按钮 → 主线程进入 View$PerformClick → Demo 执行 GcMemoryChurnScenario.run() → 主线程执行 GcMemoryChurnWorkload.churnMemoryOnMainThread()，大量创建对象并维护保留队列 → 堆相关线程 HeapTaskDaemon CPU 时间升高 → 当前点击消息耗时达到 3497ms 且采集时仍未结束 → 后续 UI、手势、动画和 Watchdog 消息排队超过 3 秒 → SDK 生成疑似 ANR 报告。

最终结论模板

本次 ANR 的根因是主线程执行 GC / 内存抖动逻辑。
证据链如下：
1. attribution.primary = CURRENT_MESSAGE_SLOW，说明 SDK 判断当前主线程消息执行过慢。
2. mainThread.current.callbackClass = android.view.View$PerformClick，说明问题发生在按钮点击回调。
3. mainThread.current.wallMs = 3497ms，endUptimeMs = null，说明采集时点击消息已执行 3.497 秒且尚未结束。
4. mainThread.stackFrames 指向 GcMemoryChurnWorkload.churnMemoryOnMainThread 和 GcMemoryChurnScenario.run，定位到 Demo 的 GC / 内存抖动场景代码。
5. threadCpu.topThreads 中 HeapTaskDaemon CPU = 3250ms，支持报告窗口内存在明显 GC / 堆维护压力。
6. barrierEvidence 与 binderBlock 均未给出主因证据，因此不应归因为 Sync Barrier、nativePollOnce 或 Binder 阻塞。

修复方案

问题点 修复建议 验收方式
主线程大量对象分配 把批量对象创建、列表构建、解析、压缩、图片处理等内存密集逻辑移到后台线程；主线程只接收最终结果并做轻量 UI 更新。 同一操作下 mainThread.current.wallMs 降到阈值以下，主线程栈不再出现大量分配场景代码。
短时间创建大量临时对象 复用对象、分批处理、减少中间集合、避免在循环中频繁创建大数组或大字符串。 HeapTaskDaemon 不再长期排在 CPU Top 第一；logcat 中 GC 频率下降。
一次点击内完成全部工作 把大任务拆成多批，每批结束后让出主线程；必要时显示进度或禁用按钮防重复触发。 Pending 队列中 UI、动画、Watchdog 消息不再持续积压超过 3 秒。
缺少线上防护 为内存密集入口增加耗时监控、采样日志、对象数量限制和降级策略；低端机或低内存状态下减少一次性处理规模。 线上报告中同类 event 的数量下降，并且 environmentSnapshot.memory 与业务参数可用于回溯。

给新人看的排查口诀

  • 先看 attribution.primary，只拿方向，不直接下结论。
  • 再看 mainThread.current，确认当前卡住的是哪条主线程消息。
  • 真正定责看 mainThread.stackFrames 和 stackSamples，本次就是 GcMemoryChurnWorkload。
  • 看到队列里有疑似 Barrier 时，要继续看 barrierEvidence，本次专项证据不支持 Barrier 泄漏。
  • 看到 HeapTaskDaemon CPU 很高时，要结合业务栈判断是否存在大量分配和 GC 压力。
